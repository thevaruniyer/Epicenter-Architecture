1. Purpose

This doctrine defines the mandatory UI and UX principles for Epicenter Education V1.

It governs every V1:

  • Screen
  • Page
  • Dashboard
  • User flow
  • Navigation pattern
  • Form
  • Component
  • Interaction
  • Status
  • Message
  • Empty state
  • Loading state
  • Error state
  • AI-assisted experience
  • Counsellor experience
  • Student experience
  • Administrator experience

This is not an inspiration document. It is the design decision system for the product.

All frontend work must follow it. A visually attractive screen that violates this doctrine is not an approved Epicenter Education screen.

When a design decision is unclear, use this doctrine before:

  • Personal taste
  • Default component-library styling
  • An older storyboard treatment
  • Developer convenience
  • A generic SaaS convention
  • A pattern copied from another school product

2. V1 Product Boundary

Epicenter Education V1 is a shared workspace for one school’s college counselling team and its Grade 11 and Grade 12 students.

V1 has three active roles:

  1. Administrator
  2. Counsellor
  3. Student

The product is designed around a continuously updated student workspace containing the student’s profile, roadmap, tasks, shortlist, applications, documents, notes, academic history, deadlines, and progress.

The Student is the primary product object.

Every task, note, roadmap milestone, university choice, application, document, risk, and update must remain connected to the relevant student.

The product must not become a collection of disconnected modules.

2.1 V1 design boundary

This doctrine applies only to screens and flows approved for V1.

It does not authorise a designer or developer to introduce a new:

  • User role
  • Navigation area
  • Dashboard
  • Workflow
  • Integration
  • Reporting surface
  • Communication channel
  • Mobile experience
  • Feature module

A new feature requires an approved product-scope change before it receives a UI.

2.2 V1 device boundary

V1 is designed for laptop and desktop browsers.

No dedicated mobile or tablet interface is required in V1.

The product must still remain usable at 200% browser zoom and at common laptop resolutions.

3. Source-of-Truth Hierarchy

When project documents conflict, use the following order:

  1. Epicenter Education PRD V1 defines the problem, V1 roles, permissions, product scope, goals, and non-goals.
  2. V3 Counsellor and Student Flow Storyboards define approved screen structure, fields, flow sequence, and role-specific journeys.
  3. AI Integrations Specification and AI Integration Flow Plan define approved AI behaviour, triggers, review rules, visibility, and provenance.
  4. This UI/UX Doctrine defines visual identity, hierarchy, interaction quality, accessibility, component behaviour, and state communication.
  5. The UI Inspiration folder is the mandatory visual reference library for translating the approved flows and this doctrine into real screens.
  6. The shared design system implements the doctrine through tokens and reusable components.
  7. Individual screen files and frontend code must comply with all layers above.

3.1 Conflict rules

  • If this doctrine conflicts with an old storyboard’s colours, typography, spacing, surfaces, or interaction styling, this doctrine wins.
  • If this doctrine conflicts with the approved order or purpose of a V3 flow, the V3 flow wins unless the Product Owner changes that flow.
  • The UI Inspiration folder guides visual execution but does not override the PRD, approved V3 flows, permissions, accessibility requirements, or this doctrine.
  • If a proposed screen is absent from the approved V1 flows, it is not added merely because it may be useful later.
  • A component-library default never overrides the Epicenter design system.

3.2 Mandatory UI Inspiration reference

CLAUDE CODE BUILD RULE

Do not design a screen from memory or from generic SaaS defaults.

Before building or revising any frontend screen, open the UI Inspiration folder, identify the closest relevant example, and compare the implementation against that example throughout the build.

Claude Code must treat the UI Inspiration folder as an active working reference, not as a folder that is reviewed once and then ignored.

For every screen or flow, Claude Code must:

  1. Open the relevant files in UI Inspiration before writing the screen.
  2. Identify which example most closely matches the page type, user role, and interaction being built.
  3. Use the example to guide composition, page proportions, spacing, typography scale, field placement, card treatment, glassmorphic surfaces, navigation behaviour, interaction rhythm, and visual density.
  4. Translate the reference into Epicenter Education using the colours, Satoshi typography, semantic states, accessibility rules, and component rules in this doctrine.
  5. Re-open the reference after the first implementation pass and compare the screen side by side before considering it complete.
  6. Name the reference file or files used in the implementation notes or pull-request summary.

Login and onboarding are fidelity-reference pages

The Login and Student Onboarding experiences must be built directly from the closest approved examples stored in UI Inspiration.

For these pages, “reference” means close visual and interaction replication of the approved example’s:

  • Overall composition and alignment
  • Content hierarchy
  • Field and control placement
  • Step progression and progress treatment
  • Spacing and negative space
  • Card and glass-surface proportions
  • Typography scale and visual rhythm
  • Primary-action placement
  • Transition behaviour

Claude Code should not merely create a generic login page or a generic onboarding wizard that happens to use Epicenter colours. It must begin with the approved visual example and adapt it faithfully to Epicenter Education’s product copy, flows, data, and doctrine.

What must not be copied

Do not copy:

  • Unrelated product features
  • External brand names or logos
  • Incorrect copy or data
  • Decorative assets that are not approved for Epicenter Education V1
  • Interaction patterns that conflict with the approved flow, privacy rules, or accessibility standards

If several UI examples conflict, use the example closest to the exact screen type and approved V1 flow. If the correct interpretation remains unclear, stop and ask the Product Owner rather than inventing a new pattern.

4. Product Experience Statement

Epicenter Education should feel:

  • Calm
  • Intelligent
  • Aspirational
  • Trustworthy
  • Minimal
  • Premium
  • Human
  • Purposeful

Epicenter Education must never feel:

  • Corporate
  • Bureaucratic
  • Institutional
  • Childish
  • Overly gamified
  • Visually loud
  • Mechanically cold
  • Like a generic school ERP
  • Like a spreadsheet placed inside cards
  • Like an analytics dashboard with no clear action

The interface should reduce anxiety while increasing clarity.

The user should feel that the product understands the complexity of university applications without transferring that complexity into the interface.

5. Foundational UX Principles

5.1 Student journey first

The student journey is the central organising principle.

Information should be found in one of two ways:

  1. By entering an individual student workspace
  2. By querying across students from a counsellor portfolio view

Do not create isolated feature areas that require the user to reconstruct student context manually.

5.2 Calm through prioritisation

Calm does not mean hiding work.

Calm is created by:

  • Clear hierarchy
  • Meaningful grouping
  • Restrained colour
  • Progressive disclosure
  • Limited competing actions
  • Explicit ownership
  • Clear next steps

A calm screen may still contain significant information, but it must never feel uncontrolled.

5.3 Action before analytics

Every operational screen must help the user act.

Prioritise:

  • Students needing attention
  • Work awaiting review
  • Overdue work
  • Upcoming deadlines
  • Blocked progress
  • Current ownership
  • Clear next actions

Charts are permitted only when they support a specific decision.

Decorative charts are prohibited.

5.4 Next action clarity

Every major object should make the next action understandable.

The user should not need to infer the next step by combining several statuses.

Where relevant, display:

  • Current status
  • Current owner
  • Who acts next
  • Due date
  • Blocking issue
  • Expected outcome after completion

5.5 Progressive disclosure

Show the most decision-relevant information first.

Reveal secondary detail through:

  • Tabs
  • Expandable sections
  • Side drawers
  • Detail pages
  • Context panels
  • Tooltips for clarification only

Do not place every available field on the initial screen.

5.6 Context over navigation

Actions should happen close to the object they affect.

Examples:

  • Document feedback belongs with the document or task it supports.
  • Application requirements belong within the relevant application.
  • Notes belong in the relevant student and meeting context.
  • Task comments belong with the task.
  • Shortlist decisions belong with the university-course entry.

Do not create a permanent top-level navigation item for every object type.

5.7 Preserve history

The product exists partly to replace unreliable memory.

Important changes must not silently erase the previous state.

History is required for:

  • Notes
  • Grades and predicted grades
  • Roadmaps
  • Task status
  • Shortlist category
  • Application status
  • Documents and submitted evidence
  • Counsellor assignment
  • AI-assisted consequential content

5.8 Privacy is visible

Privacy must be enforced technically and communicated visually.

The user must know who can see what they are entering before they save it.

5.9 Accessibility is a product requirement

Accessibility is not a final-stage visual check.

A screen that cannot be operated with a keyboard, understood without colour, or used at 200% zoom is incomplete.

5.10 One system, role-appropriate experiences

Counsellors, students, and administrators use the same visual identity and component system.

Their interfaces differ in:

  • Density
  • Navigation shell
  • Language
  • Information priority
  • Level of guidance

They must not feel like unrelated products.

  • Permanent internal provenance record

8. Action Colour Hierarchy

Both primary and secondary actions use the yellow brand family, but they must not have equal visual weight.

8.1 Primary action

Use:

  • Filled #EDC001 background
  • Black label
  • Medium or bold Satoshi
  • Strongest action prominence on the page

Examples:

  • Add Student
  • Assign Roadmap
  • Save Application
  • Submit Evidence
  • Create Task
  • Confirm Shortlist
  • Save Note

There should normally be one primary action per screen or major working context.

8.2 Secondary action

Use one of:

  • Pale yellow surface with black label
  • Yellow outline with black label
  • Text action with a yellow hover/focus treatment

Secondary actions support the primary action but must not compete with it.

There may be up to two secondary actions in a major screen context.

8.3 Tertiary action

Use:

  • Black text button
  • Neutral icon button
  • Context menu item
  • Overflow menu

8.4 Destructive action

Destructive actions never use yellow.

Use calm error-red styling and explicit wording.

Destructive actions should normally be separated from the primary action area.

9.1 Typeface

Satoshi is the mandatory V1 interface typeface.

All product UI uses Satoshi.

Permitted variations include:

  • Regular
  • Medium
  • Bold
  • Italic
  • Approved differences in font size
  • Approved differences in line height
  • Approved differences in letter spacing

Do not introduce another visual typeface into V1.

A system fallback may exist only as a technical failure fallback. It is not part of the intended design.

9.2 Typography character

Typography should feel:

  • Clear
  • Contemporary
  • Intelligent
  • Premium
  • Calm
  • Human
  • Highly readable during long working sessions

9.3 Type hierarchy

Use size and weight before colour to create hierarchy.

Role Recommended treatment
Page title 28–36px, Medium or Bold
Student name/workspace title 24–32px, Medium or Bold
Section heading 20–24px, Medium
Card heading 16–18px, Medium
Default body 16px, Regular
Dense table text 14px minimum
Button label 14–16px, Medium
Form label 14px, Medium
Metadata 12–14px, Regular
Badge/status 12–13px, Medium

9.4 Typography rules

  • Default body text is 16px.
  • Essential student instructions should normally be 16px or larger.
  • Dense counsellor tables may use 14px.
  • Critical labels must not be below 14px.
  • Non-critical metadata may use 12px.
  • Avoid excessive bolding.
  • Avoid long uppercase text.
  • Uppercase is permitted only for short metadata or eyebrow labels.
  • Italics are used sparingly for reflective prompts, supporting context, or quoted content.
  • Long text should use comfortable line height between 1.45 and 1.65.
  • Avoid overly wide text measures. Long-form content should generally stay within 60–80 characters per line.

10.1 Base spacing system

Use a 4px base spacing unit.

Preferred increments:

  • 4px
  • 8px
  • 12px
  • 16px
  • 20px
  • 24px
  • 32px
  • 40px
  • 48px
  • 64px

Avoid arbitrary spacing values unless required by a specific optical correction.

10.2 Layout character

The product should feel spacious, but not empty.

Use whitespace to:

  • Separate unrelated information
  • Establish hierarchy
  • Create reading rhythm
  • Reduce cognitive load
  • Preserve calm

Do not use excessive whitespace that forces operational users to scroll unnecessarily.

10.3 Page width

  • Counsellor pages may use broad layouts suitable for working tables and queues.
  • Student pages should use more controlled content widths and stronger visual focus.
  • Form content should not stretch unnecessarily across the full viewport.
  • Long-form editing should use a readable column width.

10.4 Grid

Use a consistent 12-column desktop grid or an equivalent documented layout system.

Common patterns:

  • Counsellor dashboard: 3-column operational grid
  • Student dashboard: main overview plus fixed right-side To-Do region
  • Detail pages: primary content plus contextual side panel
  • Forms: single focused column or grouped 2-column layout where comparison is useful

11.1 Corners

Use rounded corners throughout V1.

Recommended radius scale:

Token Value Use
radius-sm 8px Compact controls, chips
radius-md 12px Inputs, buttons, standard cards
radius-lg 16px Main cards, panels
radius-xl 24px Large summary surfaces
radius-pill 999px Pills, tabs, compact statuses

Rounded corners should feel refined, not toy-like.

11.2 Containers

Frequent containers are part of the visual identity.

Use containers to group information with a common purpose.

Do not:

  • Put every line inside a card
  • Nest several cards without clear hierarchy
  • Use cards to fill visual space
  • Split one workflow into unrelated floating boxes

11.3 Borders

Borders create soft surface separation.

Default borders should be:

  • Thin
  • Low contrast
  • Warm neutral
  • Consistent

Strong black borders are reserved for:

  • Privacy emphasis
  • Focus states
  • High-importance selected objects
  • Specific black semantic treatments

11.4 Shadows

Shadows must be subtle.

Default cards should rely primarily on:

  • Surface difference
  • Border
  • Spacing

Use shadows for:

  • Floating menus
  • Modals
  • Drawers
  • Drag states
  • Elevated glass surfaces
  • Temporary overlays

Avoid deep, dramatic, or coloured shadows.

11.5 Icons

Icons must be:

  • Minimal
  • Consistent
  • Recognisable
  • Optically balanced
  • Accessible

Use one icon family throughout V1.

Pair icons with text when the meaning is not universally obvious.

Icons are not decoration.

11.6 Illustrations

Illustrations are not part of V1.

Do not introduce decorative illustrations, mascots, or character art.

Use typography, surfaces, data, and restrained iconography to create visual interest.

12. Glassmorphism Doctrine

Glassmorphism is a controlled part of the Epicenter Education visual language.

It should communicate selected state, layering, or premium focus.

12.1 Approved uses

Use glassmorphism for:

  • Selected navigation
  • Floating utility controls
  • Context panels
  • Command overlays
  • Important summary surfaces
  • Modal or drawer surfaces
  • Focused dashboard highlights

12.2 Base glass token

A typical V1 glass surface may use:

  • Background: rgba(255, 255, 255, 0.72)
  • Backdrop blur: 12px–18px
  • Border: 1px solid rgba(0, 0, 0, 0.08)
  • Shadow: 0 8px 30px rgba(0, 0, 0, 0.06)
  • Black text
  • Yellow selected indicator where appropriate

These are starting values and should be implemented as shared tokens.

12.3 Selected navigation

Selected navigation uses:

  • Glass surface
  • Black text
  • Stronger font weight
  • Yellow indicator, underline, dot, or edge accent
  • Clear focus state
  • Sufficient contrast without blur

12.4 Restrictions

Glassmorphism must not:

  • Be used on every card
  • Reduce text contrast
  • Sit over visually noisy backgrounds
  • Make active and inactive states ambiguous
  • Obscure keyboard focus
  • Replace structural hierarchy
  • Become decorative spectacle

Where blur is unsupported or accessibility suffers, use an opaque fallback with the same hierarchy.

13. Motion Doctrine

Hover or focusSubtle functional transition

Motion clarifies state and preserves context. It does not decorate the workflow.

Motion is subtle and functional.

13.1 Approved motion

Use motion for:

  • Drawer opening and closing
  • Accordion expansion
  • Tab changes
  • Status transitions
  • Progress changes
  • Save confirmation
  • Hover and focus feedback
  • Context-preserving page transitions
  • Reordering where the movement communicates the new state

13.2 Motion timing

Interaction Duration
Micro-interaction 120–180ms
Tab or compact panel 160–220ms
Drawer or modal 180–240ms
Page-level transition Maximum 300ms

Use natural easing.

13.3 Motion restrictions

Do not use:

  • Bounce effects
  • Large-scale parallax
  • Decorative looping animation
  • Delayed animation that blocks action
  • Celebration confetti
  • Motion that makes professional work feel playful
  • Animation that causes layout shift

Respect prefers-reduced-motion.

15.1 Desired character

The counsellor experience should feel:

  • Calm
  • Professional
  • Intelligent
  • Controlled
  • Spacious
  • Context-rich
  • Efficient
  • Trustworthy

It should not feel:

  • Like a crowded control room
  • Like a CRM sales dashboard
  • Like an endless spreadsheet
  • Like an analytics report
  • Like a colourful student portal

15.2 First ten seconds

Within ten seconds of logging in, a counsellor should understand:

  1. What is on today
  2. Which students need attention
  3. Which student work is overdue
  4. What is waiting for counsellor review
  5. What meaningful progress has happened

The interface should include the approved V1 calendar or today-schedule surface without implying unapproved external functionality.

15.3 Dashboard hierarchy

The counsellor dashboard should prioritise:

  1. Today / Calendar

    • Meetings today
    • Relevant deadlines
    • Time-sensitive work

  2. Students needing attention

    • Risk or stalled progress
    • No recent activity
    • Meaningful exceptions

  3. Overdue student work

    • Student
    • Task
    • How overdue
    • Current owner
    • Direct action

  4. Awaiting counsellor review

    • Evidence
    • Documents
    • Notes
    • Shortlist changes
    • Application work

  5. Progress overview

    • High-level cohort or caseload progress
    • No decorative charts
    • Every summary should link to the relevant records

15.4 Dashboard behaviour

Dashboard items must be actionable.

Each item should deep-link to the exact:

  • Student
  • Task
  • Document
  • Application
  • Note
  • Review panel
  • Calendar item

Do not link every alert merely to the student overview.

15.5 Counsellor density

The counsellor interface uses a calm, spacious layout with progressive disclosure.

It should support professional information density through:

  • Tables
  • Compact rows
  • Filters
  • Saved views where approved
  • Bulk selection where approved
  • Side drawers
  • Expandable detail
  • Sticky headers

It must not display all details simultaneously.

15.6 Counsellor language

Use concise, professional language.

Examples:

  • Awaiting your review
  • 3 days overdue
  • Student action required
  • Updated 2 hours ago
  • Review submission
  • View student context

Avoid:

  • Fun slogans
  • Excessive encouragement
  • Gamified wording
  • Unclear status language

  • Personal

  • Application requirements
  • Forms included in the approved V1 flow
  • Document submission requests

The student should not need to inspect multiple sections to understand what they need to do next.

16.4 Student motivation

  • Explicit ownership

  • Meaningful milestone completion
  • Clear counsellor feedback

Do not use:

  • Streaks
  • Leaderboards
  • Competitive ranking
  • Artificial rewards
  • Confetti
  • Excessive celebration

  • Due Friday
  • Send to your counsellor
  • Changes requested
  • Your counsellor reviewed this

Avoid patronising language.

16.6 Student disclosure

  • Their applications

35.1 AI review

Consequential AI-assisted content must be:

  • Previewed
  • Editable
  • Explicitly approved or saved
  • Logged
  • Attributed as AI-assisted internally

35.2 Passive AI surfaces

Approved passive AI surfaces may be:

  • Viewed
  • Acted upon
  • Dismissed

They do not require a save action because they do not directly publish content.

Examples include approved counsellor-only:

  • Daily digest
  • Risk flags
  • Stalled-task alerts

35.3 Grounding

Any AI output that states a:

  • Date
  • Count
  • Name
  • Status
  • Deadline
  • Grade
  • Task state

must use real product data.

The model may phrase the result, but must not invent the fact.

35.4 Pull over push

AI should not interrupt the user unnecessarily.

Prefer:

  • On-demand actions
  • Quiet dashboard surfaces
  • Contextual suggestions
  • Dismissible insights

Avoid:

  • Repeated pop-ups
  • Forced AI flows
  • Automatic publishing
  • External AI reminders
  • Surprise changes

35.5 Student visibility

Students should see approved counsellor feedback.

Students should not automatically see:

  • Internal AI drafting
  • Raw AI output
  • Counsellor-only risk analysis
  • Internal AI summaries
  • Unapproved feedback

35.6 AI provenance

The system must retain internal provenance for every AI-assisted action.

Record:

  • Feature
  • Student
  • Original input
  • AI output
  • Final human-approved content
  • Reviewer
  • Review time
  • Whether edits were made

35.7 AI marker

Use a minimal black AI marker.

The marker should feel factual, not magical.

Counsellors see the AI marker on AI-generated or AI-assisted internal content.

Where an approved V1 flow intentionally presents final counsellor feedback to a student without exposing the internal drafting process, the student UI shows the approved feedback as counsellor feedback while the system retains provenance in the audit record.

35.8 AI processing state

When AI is processing, show:

  • What is happening
  • That the user may continue where possible
  • How to cancel if supported
  • Error and retry behaviour

Avoid vague “Thinking…” states with no context.

  • Disappear
  • Become unreachable

Respect prefers-reduced-motion.

Remove non-essential transitions while preserving state clarity.

  • Clearly visible
  • High contrast
  • Consistent
  • Not removed for aesthetic reasons

  • Specific
  • Human
  • Respectful
  • Action-oriented
  • Free from internal jargon

38.1 Counsellor copy

Counsellor copy should be concise and professional.

Prefer:

  • Review submission
  • Student action required
  • Waiting for student
  • Due in 3 days
  • Updated yesterday
  • View student context

38.2 Student copy

Student copy may be warmer and more encouraging.

Prefer:

  • Your next step
  • Continue your draft
  • Your counsellor has reviewed this
  • Due Friday
  • Changes requested
  • Send to your counsellor

38.3 Button labels

Buttons describe the action.

Use:

  • Save Note
  • Submit Evidence
  • Add University
  • Convert to Application
  • Request Changes
  • Assign Roadmap

Avoid:

  • Continue, where the outcome is unclear
  • Submit, without an object
  • Process
  • Execute
  • Initiate
  • OK, except in simple acknowledgements

38.4 Status copy

Avoid vague words such as:

  • Pending
  • Processing
  • Incomplete
  • Attention

without explanation.

PART XIII — GOVERNANCE

40. Doctrine Authority

The Epicenter Education Product Owner is the final approver for:

  • Doctrine changes
  • Design exceptions
  • Navigation changes
  • New semantic treatments
  • New shared component patterns
  • Deviations from approved V1 flows

Designers and engineers may propose exceptions.

They may not approve their own exceptions.

41. Doctrine Versioning

This doctrine changes only when the Product Owner explicitly requests or approves an iteration.

41.1 Patch version

Example: 1.0.1

Use for:

  • Typographical fixes
  • Clarification
  • Additional examples
  • Non-behavioural correction

41.2 Minor version

Example: 1.1

Use for:

  • New component rule
  • Refined interaction pattern
  • Additional approved V1 flow guidance
  • Updated token values that preserve the identity

41.3 Major version

Example: 2.0

Use for:

  • Fundamental visual identity change
  • Navigation change
  • New interaction philosophy
  • Major role change
  • Major scope change

Every update records:

  • Date
  • Approver
  • Summary
  • Affected screens
  • Required implementation changes

42. Exception Process

Every exception must document:

  • Rule being overridden
  • Reason
  • Affected screen or component
  • User benefit
  • Risk
  • Approver
  • Approval date
  • Review or expiry date

Exceptions must not become permanent patterns through repetition.

At review, an exception must be:

  • Removed
  • Renewed
  • Or incorporated into a doctrine update

PART XIV — DESIGN REVIEW CHECKLIST

44. Purpose

  • Is the user problem clear?
  • Is the screen’s purpose immediately understandable?
  • Is the primary action obvious?
  • Does the screen help the user act or decide?

45. Hierarchy

  • Is the most important information shown first?
  • Is progressive disclosure used?
  • Is the page calm without becoming empty?
  • Are unrelated elements visually separated?
  • Are related elements grouped?

46. Role fit

  • Is the screen designed for Administrator, Counsellor, or Student?
  • Is the density appropriate?
  • Is the student experience guided?
  • Is the counsellor experience professional?
  • Does the administrator experience stay functional and restrained?

47. Navigation

  • Does the screen follow the fixed V1 shell?
  • Is the active location obvious?
  • Can the user return to the previous context?
  • Does a portfolio item deep-link to the exact relevant object?
  • Has an unnecessary top-level destination been introduced?

48. Actions

  • Is there one primary action?
  • Are there no more than two secondary actions?
  • Are destructive actions separated?
  • Are labels explicit?
  • Does consequential action require confirmation?

49. Workflow

  • Is the current owner visible?
  • Is the current status visible?
  • Is the last action visible where relevant?
  • Is the last updated time visible?
  • Is the next actor visible?
  • Is the due date visible?
  • Is the outcome after completion clear?

50. Privacy

  • Is the audience explicit?
  • Does private content use a lock icon?
  • Is visibility clear before saving?
  • Could an unauthorised user infer hidden private content?
  • Are previews, search, and activity feeds safe?

51. AI

  • Is AI provenance visible to the correct reviewing user?
  • Does consequential AI content require human confirmation?
  • Is factual output grounded in real data?
  • Is internal analysis hidden from students?
  • Is provenance retained in the system?

52. Accessibility

  • Can the screen be fully used by keyboard?
  • Is focus visible?
  • Does contrast pass?
  • Is meaning understandable without colour?
  • Does the layout work at 200% zoom?
  • Is reduced motion supported?
  • Are labels available to screen readers?

53. States

  • Is the empty state designed?
  • Is the loading state designed?
  • Is the error state designed?
  • Is the disabled state designed?
  • Is the success state designed?
  • Is work preserved after failure?
  • Is recovery clear?

54. System consistency

  • Does the screen use Satoshi?
  • Does it use the approved colour tokens?
  • Does it use shared components?
  • Does selected navigation use the approved glass treatment?
  • Was the closest relevant file in UI Inspiration opened and used during the build?
  • Does the final screen preserve the composition, visual rhythm, and interaction treatment of that reference where appropriate?
  • For Login or Onboarding, does the screen closely replicate the approved UI example rather than a generic alternative?
  • Has a one-off pattern been introduced?
  • Does wording match shared status language?

55. Final Decision Priority

When principles conflict, use this order:

  1. Privacy and student safety
  2. Accessibility
  3. Clarity
  4. Task completion
  5. Workflow transparency
  6. Context
  7. Efficiency
  8. Consistency
  9. Brand expression
  10. Visual elegance

A beautiful interface that hides ownership, weakens privacy, obscures status, or makes the next action unclear is a failed Epicenter Education interface.

56. Definition of Done for V1 UI/UX

A V1 screen is complete only when:

  • It follows an approved V1 flow
  • It uses the fixed role navigation shell
  • It uses Satoshi
  • It uses approved tokens
  • It has one clear primary action
  • It communicates ownership and status
  • It defines privacy behaviour
  • It defines AI behaviour where applicable
  • It includes empty, loading, error, disabled, and success states
  • It passes WCAG 2.2 AA checks
  • It works with keyboard navigation
  • It works at 200% zoom
  • It uses shared components
  • The closest relevant example in UI Inspiration was opened before implementation and reviewed again after the first pass
  • The implementation notes or pull-request summary name the UI Inspiration reference file or files used
  • Login and Onboarding closely reproduce their approved UI examples while following Epicenter tokens, copy, flows, and accessibility requirements
  • It passes the doctrine review checklist
  • Any exception is formally documented and approved

57. V1 Design Summary

Epicenter Education V1 is a calm, intelligent, aspirational, and trustworthy desktop product.

Its visual system is built around:

  • #EDC001 Epicenter Yellow
  • #FDFDFD primary surfaces
  • #000000 text and secondary brand identity
  • Satoshi typography
  • Rounded containers
  • Soft borders
  • Very subtle shadows
  • Restrained glassmorphism
  • Minimal icons
  • No illustrations
  • Subtle functional motion

Its experience is built around:

  • One continuously updated student record
  • A professional counsellor portfolio and student workspace
  • A guided student journey with an immediate right-side To-Do List
  • Clear deadlines and ownership
  • Explicit privacy
  • Human-confirmed consequential AI
  • Actionable dashboards
  • Progressive disclosure
  • Accessible, consistent interaction patterns
  • Continuous visual reference to the approved examples in UI Inspiration

This doctrine is the V1 source of truth for all Epicenter Education frontend design and implementation.
