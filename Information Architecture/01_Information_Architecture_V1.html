Epicenter Education — Information Architecture (V1)

A working architecture document for a unified college-counselling platform. Organised around the student's journey, not a list of features. Companion to 02_Information_Architecture_Explanation.md.

1Architecture Summary

Central organising principle: everything attaches to the student's journey — tasks, meetings, notes, documents, shortlist entries, applications, recommendations, risks. Nothing is a standalone module; everything is either found by drilling into one student, or by querying across all students.

Primary object: Student. The secondary anchor is the Application — the point where requirements, documents, recommendation status and transcript status converge for one university/course attempt.

Key decision — four shells, not seven apps. Counsellor and Head of Counselling share one shell (scope differs, not structure). Student and Parent share one shell (Parent is a permission-filtered subset). Teacher/Recommender gets a single-purpose portal. School Admin gets a settings-only shell. A Platform Admin console exists separately and is out of scope here.

Portfolio vs. student view: not two data sets — two lenses on the same objects. Portfolio = cross-student query ("every overdue task across 35 students"). Student Workspace = single-student query ("everything about Ananya"). Every portfolio row must click straight into the exact tab/task/document it refers to.

Dashboards are actionable queues, not charts — "what do I do, where do I click," never "how are we trending." Aggregate charts live in Reports (Post-MVP).

Tasks and roadmaps: one task model, viewable five ways (portfolio queue, student roadmap, inside an application, inside a milestone, on a dashboard) — never five disconnected systems.

Shortlist → application: a University + Course pairing moves through Suggested → Shortlisted → Approved → Application → Submitted → Outcome. Conversion is a deliberate 2-click action, not automatic.

Permission philosophy: narrowest access that lets a role do their job. Counsellors see everything about every student (flat, per PRD). Students see everything about themselves except what's explicitly private/confidential. Parents and teachers get a narrow, purpose-built slice — never a cut-down copy of the counsellor's view.

2Recommended Global Navigation, by Role

Not every role needs a distinct navigation structure. See §1 for the four-shell decision. Below is the final top-level nav per role, with rationale.

Counsellor

  1. Home — overdue tasks, today's meetings, essays to review, what changed since last login.
  2. Students — the portfolio; entry point into any Student Workspace.
  3. Applications — cross-student command centre: deadlines, blocked apps, missing recs/transcripts.
  4. Universities — shared reference library (manual entry, V1).

Head of Counselling

  1. Same shell as Counsellor, scope = all students, not just assigned.
  2. Team — counsellor list, caseload sizes, reassignment.
  3. Reports Post-MVP — cohort/bottleneck reporting.

Student

  1. Home — "what do I need to do next."
  2. My Roadmap — roadmap + tasks combined.
  3. My Universities — shortlist + applications + offers, one pipeline.
  4. Documents — own library.
  5. Meetings & Notes — shared notes only.
  6. Profile — incl. activities/achievements.

Parent Phase 2

  1. Home — high-level progress, major deadlines only.
  2. Universities — shortlist + offers, no application-internal detail.
  3. Decisions — deposits, scholarship conditions.

Excluded: Documents, Meetings & Notes, Roadmap detail, Recommendations.

Teacher / Recommender Post-MVP

  1. Recommendation Requests — effectively the entire portal: a queue, expandable to questionnaire + upload.

No access to caseload, roadmap, notes, or anything else.

School Administrator

  1. School Profile — incl. white-label branding.
  2. Users & Roles
  3. Counsellor Assignments
  4. Templates — roadmap & task.
  5. Audit Logs
  6. Post-MVP Integrations, Risk Rules, Data Import/Export
[ASSUMPTION] Platform Administrator is a separate internal tool (school list, onboarding, platform settings) not covered in this school-facing IA — only relevant once more than one school is live.

4Student Workspace Structure

Consolidated to 8 tabs from the ~15 candidate sections in the original brief — see §10 for what was merged and why.

Overview MVP

Shows: completion, grade, counsellor, majors/countries, roadmap %, deadlines, overdue, risks, next action.

Access: Counsellor, Head, Student (self), Admin (read).

Profile MVP

Shows: personal/contact/parent info, preferences, majors, countries, budget notes, career interests, activities & achievements.

Access: Counsellor (edit), Student (edit own), Admin (create).

Academics MVP

Shows: subjects, grades/predicted grades per checkpoint (G11 HYE/Finals → G12 Boards), tests planned/actual.

Access: Counsellor (edit), Student (view), Admin (read).

Roadmap MVP

Shows: milestones (one continuous G11→G12 roadmap), tasks, status, feedback.

Access: Counsellor, Student (both edit).

Universities MVP

Shows: shortlist entries → applications → offers/decisions, one pipeline; rec/transcript status strips inside each application.

Access: Counsellor, Student, Parent (Ph.2, outcomes only).

Documents MVP

Shows: full library, category, version, review status, reverse-links showing reuse.

Access: Counsellor, Student.

Meetings & Notes MVP

Shows: meeting history, shared notes, private notes (counsellor-only sub-view), student updates, follow-ups.

Access: Counsellor (all types), Student (shared + own only).

Activity History MVP

Shows: chronological log across every tab — "what changed since last time."

Access: Counsellor, Head, Admin (audit).

Deliberately not separate tabs: Tests (→ Academics), Activities/Achievements (→ Profile), Shortlist/Applications/Offers (→ one Universities pipeline), Recommendations & Transcript Requests (→ status strips inside an Application until Post-MVP).

Shortlist → Application → Outcome pipeline

Shortlist Entry→ Suggested→ Shortlisted→ Approved→ (2-click convert) → Application→ Submitted→ Outcome / Offer

Key cardinalities

  • One School → many Students, Users, Academic Years.
  • One Student → many Counsellor Assignments over time (a history table, not a single field — preserves who was responsible for what, when).
  • One Student → one active Roadmap (cloned from a Roadmap Template), spanning G11 + G12 as one continuous record — not two objects that hand off.
  • One Roadmap → many Milestones; one Milestone → many Tasks.
  • Task has a polymorphic parent: Milestone, Student (ad hoc), Application Requirement, or Meeting (follow-up) — one model, many views.
  • One Student → many Meetings; a Meeting may have one Shared Note. Private Notes are their own type, linked to Student (+ optional Meeting).
  • One University → many Courses. A Shortlist Entry = Student + University + Course + Round — this is what allows two courses at one university without conflict.
  • A Shortlist Entry → 0 or 1 Application. No conversion = "considered but never applied," a valid end state.
  • One Application → many Application Requirements (essay, test score, recommendation, transcript, portfolio, interview).
  • An Application Requirement ↔ many Documents (many-to-many) — stored once in the student's library, linked everywhere, never copied.
  • A Recommendation Request belongs to the Student (not one Application) and links to many Applications it supports — this is what makes "every application blocked by a missing recommendation" a simple query.
  • A Transcript Request follows the same pattern as Recommendation Request.
  • A Risk belongs to a Student and may have many Interventions logged against it.
  • An Application → 0 or 1 Offer; an Offer may reference a Scholarship. Accepting one Offer creates one Final Decision at the Student level.
  • Comment and Notification are polymorphic — they attach to whatever object triggered them.
  • Audit History is system-generated on writes — never user-created.

8Risks and Open Questions

Architecture risks

The Applications Command Centre is the highest-value screen but also the most complex to build (aggregates roadmap tasks, requirements, recommendations, transcripts). Under-scoping it undercuts the platform's core differentiation vs. spreadsheets.

Folding Recommendations and Transcript Requests into status strips is right for V1, but may need to become full tabs later — decide the trigger condition now.

Permission risks

Flat "all counsellors see all students" works at 3 counsellors but has no defined stopping point.

Recommendation-letter confidentiality mechanism is unresolved — do not build Documents/Recommendation objects until settled, or a rebuild is likely.

Data duplication risks

Documents must be modeled as link-not-copy from day one — retrofitting later is expensive.

Grades feed shortlist eligibility — one source of truth, read in two places, not two entry points.

Navigation risks

Promoting Tasks/Roadmaps/Shortlists back to top-level nav (a common ask) risks re-fragmenting the product — see §10.

Teachers/Parents must never get a "lite counsellor nav" — narrow, purpose-built screens only.

Scalability risks

Manual entry for grades/universities won't scale past a handful of schools — an accepted V1 trade-off, not an oversight.

One continuous roadmap per student works for one curriculum; multi-curriculum schools will likely need country-specific template variants — not yet designed.

Product-scope risks

Sensitive private notes + eventual financial data create real "generic school CRM" scope-creep risk — test every new feature against the counsellor's daily questions.

No instrumentation yet for whether the product actually beats spreadsheets (PRD Open Q11) — worth measuring informally from day one.

Questions to answer before user-flow design begins
  1. Exact confidentiality enforcement mechanism for recommendation letters?
  2. Is shortlist/grade diff-history coming to V1, or staying deferred?
  3. Does the Applications Command Centre need real-time updates, or is daily/nightly refresh acceptable?
  4. Will a second school onboard before V1 ships (affects whether School Settings needs real multi-tenant config now)?
  5. Does "Assigned to" need full audit-trail history from day one, or is a current-value field acceptable for the pilot?

10Critical Review — What Changed and Why

Top-level items cut after review

The brief's candidate top-level list (Tasks, Roadmaps, Shortlists, Applications, Documents, Meetings, Recommendations, Transcript Requests — nine items) would produce an unmanageable 13-item nav once Dashboard/Students/Reports/Settings are added, and would re-fragment the product into disconnected modules. Only Applications earned a standalone slot, as the one cross-student queue with clear standing daily value.

Overlap identified and resolved

Shortlist, Applications, and Offers/Decisions were three candidate sections describing three stages of the same object lineage. Keeping them separate risked the same University+Course pair appearing three different places as it progressed. Merged into one Universities tab with an internal pipeline.

Same data appearing twice — caught and fixed

Tests were a candidate standalone tab but overlap with Academics for eligibility judgments — folded in. Activities/Achievements are about who-the-student-is, not the application cycle — folded into Profile.

Where users may get confused

Recommendation status (student-visible) sitting next to a final letter (student-invisible) on the same object can read as a bug rather than a deliberate boundary — the UI must communicate why something is locked, not just hide it. Flagged for user-flow design.

Features that are filters, not pages

"At-risk students," "not recently contacted," "overdue tasks," "essays awaiting review," "missing documents" are all filtered views of Students or Applications — not separate pages or data.

Actions made contextual, not navigational

Add a note, assign a task, convert shortlist to application, request a transcript — all implemented as buttons inside the Student Workspace, not destinations you navigate to independently.

Where the workspace still risks too many tabs

Universities is doing the most work of any tab. If application volume per student grows (10+ schools), it may need its own list+detail layout rather than one scrolling view — a layout decision for user-flow design, not a structural one.

Administrative complexity delayed on purpose

Integrations, Risk Rules, Notification Rules, Data Import/Export are real needs eventually, but building configuration UI for features that don't exist yet (single school, manual entry, no notifications) is wasted scope. Delayed to Post-MVP/Future.

Where this could drift into a generic school CRM

The full object list from the brief could expand into a general SIS if every configuration option is exposed. Guardrail applied throughout: if a feature doesn't serve a counsellor's daily question or a student's "what's next," it gets no permanent nav slot.

Where the architecture must clearly beat spreadsheets and email

The Applications Command Centre and unified Activity History are the two places doing that job — a spreadsheet cannot show "what changed since I last looked" or "everything blocked across my caseload, live" without manual maintenance. These were treated as first-class, not an afterthought.
