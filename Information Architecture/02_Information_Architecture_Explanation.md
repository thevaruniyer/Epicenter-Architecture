## 1. Architecture Summary (Deliverable 1)

**Central organising principle.** The product is organised around **the student's journey**, not around a list of features. Every module — tasks, meetings, notes, documents, shortlist entries, applications, recommendations, transcripts, risks — attaches to a student, and usually to a specific point in that student's journey (a milestone, an application, a meeting). Nothing is designed to stand alone as a "module you visit"; everything is designed to be found either by drilling into one student, or by querying across all students.

**Primary system object.** **Student** is the anchor. Almost every other object either belongs directly to a Student (Profile, Roadmap, Meeting, Note, Risk) or belongs to something that belongs to a Student (a Task belongs to a Milestone which belongs to a Roadmap which belongs to a Student; a Document belongs to an Application Requirement which belongs to an Application which traces back to a Shortlist Entry which belongs to a Student). The secondary anchor is the **Application**, which is the object where requirements, essays, documents, recommendation status, and transcript status all converge for one university/course attempt.

**Main navigation approach.** Rather than building seven separate applications for seven roles, there are effectively **four navigation shells**, reused across roles with scope and permission differences:

1. **Counselling shell** — shared by Counsellor and Head of Counselling. Same pages, same components; the difference is data scope (assigned caseload vs. entire school) and a small number of additional items (Team, Reports) for the Head.
2. **Student/Parent shell** — Parent (Phase 2) is a permission-filtered subset of the Student's own view, not a different app.
3. **Recommender portal** — a deliberately narrow, single-purpose shell for Teachers, unrelated to the counselling shell's navigation.
4. **School Settings shell** — configuration-oriented, visible only to School Administrators (and partially to Head of Counselling for counsellor assignment).

A fifth surface, the **Platform Administrator console**, is a separate internal tool for managing schools at the multi-tenant level. It is out of scope for this school-facing IA and is only referenced briefly (see §9).

**Portfolio view vs. student view.** These are not two different data sets — they are two different **lenses on the same objects**. The Portfolio (Students list, Applications Command Centre) is a cross-student query: "show me every task overdue across my 35 students." The Student Workspace is a single-student query: "show me everything about Ananya." A counsellor should always be able to move from a row in a portfolio queue directly into the exact tab, task, or document that row refers to — this single requirement drives most of the navigation and breadcrumb design below.

**Role of the student workspace.** The student workspace is the **complete, continuously updated record** that replaces the notebook. It is intentionally the most detailed part of the product, because it is the thing that has to be better than a notebook + spreadsheet + inbox combined, or the product fails on its core value proposition.

**Role of dashboards.** Dashboards are **actionable queues, not charts**. Every dashboard item answers "what do I do next and where do I click," not "how are we trending." Aggregate charts belong in Reports (Head of Counselling, Post-MVP), not in the day-to-day Home screen.

**Role of tasks and roadmaps.** There is **one task model**, not five. A Task can be generated by a Roadmap Milestone, created ad hoc by a student or counsellor, attached to an Application Requirement, or spun out of a Meeting as a follow-up — but it is always the same underlying object, viewable from the student's Roadmap tab, from the portfolio-level task queue, or from inside the Application it relates to.

**Shortlist → application transition.** A **University + Course pairing** is the unit that moves through the pipeline: *Suggested → Shortlisted → Approved → Application → Submitted → Outcome*. A Shortlist Entry is created first (low commitment); it becomes an Application only when the student or counsellor explicitly converts it (2-click action, per the PRD). This means a student can apply to two courses at the same university (two Shortlist Entries, two Applications) without any modeling conflict.

**Permission philosophy.** Default to the **narrowest access that lets a role do their job**, expand only where the PRD or a clear workflow need justifies it. Concretely: counsellors see everything about their students (and, per the PRD's flat team model, everything about every student on the team); students see everything about themselves except what is explicitly private to the counsellor or confidential (final recommendation letters); parents and teachers get a deliberately narrow slice, not a filtered copy of the counsellor's view.

## 2. Recommended Global Navigation (Deliverable 2)

### 2.1 Counsellor

| Nav item | Why it exists |
|---|---|
| **Home** | Daily entry point. Answers "who needs me today" — overdue tasks, meetings today, essays awaiting review, what changed since last login. Not charts. |
| **Students** | The portfolio — every assigned student, searchable/filterable, the entry point into any Student Workspace. This is where "which students are falling behind" gets answered. |
| **Applications** | The cross-student application command centre — deadlines this week, blocked applications, missing recommendations/transcripts, across the whole caseload. Elevated to top-level specifically because the PRD and original vision identify this as the platform's single highest-value view during application season. |
| **Universities** | A shared, lightweight reference library of universities/courses (manually maintained in V1) used when building shortlists. Not per-student data, so it doesn't belong inside a student workspace. |
| *(profile menu)* Settings, My Account | Personal preferences only — not school configuration. |

Deliberately **not** top-level: Tasks, Roadmaps, Shortlists, Meetings, Documents, Recommendations, Transcript Requests. Each of these is either fully contextual to one student (lives inside the Student Workspace) or better represented as a filter within Home/Students/Applications than as its own permanent section. Rationale is detailed in §11 (Critical Review).

### 2.2 Head of Counselling

Same shell as Counsellor, plus:

| Nav item | Why it exists |
|---|---|
| **Students** (scope: all, not just assigned) | Needs school-wide visibility, not just a personal caseload. |
| **Team** | Counsellor list, caseload sizes, reassignment tool — answers "who's overloaded." |
| **Reports** *(Post-MVP)* | Cohort-wide, counsellor-performance, and bottleneck reporting. Excluded from MVP because the PRD explicitly rules out monthly/board-level reporting for V1. |

### 2.3 Student

| Nav item | Why it exists |
|---|---|
| **Home** | "What do I need to do next" — the single most important question a student has. |
| **My Roadmap** | Roadmap + tasks combined — for a student, the roadmap *is* their task list, organised by milestone. |
| **My Universities** | Shortlist, applications, and offers/decisions combined — one continuous pipeline from the student's point of view, not three disconnected pages. |
| **Documents** | Their own document library — upload once, see what it's attached to. |
| **Meetings & Notes** | Meeting history and shared notes only (private counsellor notes never appear here). |
| **Profile** | Personal info, preferences, activities/achievements. |

### 2.4 Parent *(Phase 2 — shown for completeness, not MVP)*

| Nav item | Why it exists |
|---|---|
| **Home** | High-level progress and major upcoming deadlines only. |
| **Universities** | Shortlist and offers — enough to be informed, not enough to see application-internal detail like essay drafts or private notes. |
| **Decisions** | Where a final decision, deposit deadline, or scholarship condition needs parent awareness or action. |

Deliberately excluded from parent nav: Documents (essay drafts), Meetings & Notes, Roadmap detail, Recommendations. Parents get outcomes and milestones, not process.

### 2.5 Teacher / Recommender *(Post-MVP — recommendation module unresolved per PRD)*

| Nav item | Why it exists |
|---|---|
| **Recommendation Requests** | The entire portal is effectively this one queue: pending requests, each expandable to the student's questionnaire/brag sheet and an upload action. |

No access to caseload, roadmap, notes, or any other part of the counselling record — a teacher's nav is intentionally a single page, not a scaled-down counsellor view.

### 2.6 School Administrator

| Nav item | Why it exists |
|---|---|
| **School Profile** | Name, branding (white-label), academic year, grade levels, curricula. |
| **Users & Roles** | Add/remove counsellors, students, teachers; assign roles. |
| **Counsellor Assignments** | Who is assigned to whom — shared with Head of Counselling's Team page. |
| **Templates** | Roadmap and task templates (school-wide defaults). |
| **Audit Logs** | Who changed what, school-wide — distinct from a single student's Activity History. |
| *(Post-MVP)* Integrations, Risk Rules, Data Import/Export | Advanced configuration, deliberately deferred (see §8). |

If the same person is both an Admin and a Counsellor, they get both shells with a switcher — not a merged, cluttered nav.

### 2.7 Platform Administrator

Out of scope for this document. **[ASSUMPTION]** This is a separate internal tool (school list, onboarding status, platform-wide settings, support tooling) that does not share UI with the school-facing product. Only relevant once more than one school is live — not needed for the single-school pilot.

## 4. Student Workspace Map (Deliverable 4)

The workspace is **8 tabs**, deliberately consolidated from the ~15 candidate sections in the brief (see §11 for what was merged and why).

| Tab | Purpose | Main information | Main actions | Roles with access | MVP? | Related sections |
|---|---|---|---|---|---|---|
| **Overview** | One-screen snapshot before a meeting or check-in | Profile completion, grade, counsellor, intended major/countries, roadmap %, upcoming deadlines, overdue work, open risks, next recommended action | Jump to any tab; mark a risk reviewed | Counsellor, Head, Student (self, simplified), Admin (read) | Yes | Pulls from every other tab |
| **Profile** | Who this student is | Personal/contact/parent info, preferences, intended majors, countries, budget/scholarship notes, career interests, activities & achievements | Edit (counsellor or student), Admin initial creation | Counsellor, Student (self), Admin | Yes | Feeds shortlist eligibility, Overview |
| **Academics** | Academic standing over time | Subjects, current/predicted grades at each checkpoint (G11 HYE/Finals, G12 Mid-Terms/HYE/Mocks/Boards), test plans & scores | Counsellor logs grade/test results | Counsellor, Student (view), Admin (read) | Yes | Drives shortlist eligibility & risk flags |
| **Roadmap** | "What's assigned and what's done" | Milestones (G11 → G12, one continuous roadmap), tasks under each milestone, ad hoc tasks, status, feedback | Assign/duplicate template, mark complete, comment, upload, request revision | Counsellor, Student | Yes | Tasks also surface in Home/Applications queues |
| **Universities** | The full shortlist-to-outcome pipeline | Shortlist entries (category, preference, eligibility), converted applications (requirements, essays/documents linked, recommendation & transcript status strip, interview notes), offers/decisions | Add/approve/categorize shortlist entry, 2-click convert to application, update status, record offer/decision | Counsellor, Student, Parent (Phase 2, outcomes only) | Yes | Documents (linked, not duplicated), Activity History |
| **Documents** | One library, reused everywhere | All files with category, version, review status, and a reverse-index of which requirements/applications reference each file | Upload, review, comment, mark version | Counsellor, Student | Yes | Referenced (not copied) from Universities/Applications |
| **Meetings & Notes** | Full conversational history | Upcoming/past meetings, shared notes, private counsellor notes (separate sub-view), student updates, follow-up tasks | Log meeting, add note (type-specific), convert note to task | Counsellor (all note types), Student (shared + own updates only) | Yes | Follow-ups become Tasks on Roadmap |
| **Activity History** | "What changed since last time" | Chronological log of edits across every tab, with actor and timestamp | Filter by type/date | Counsellor, Head, Admin (audit) | Yes | Auto-generated from every other tab |

**Not separate tabs (deliberately):** Tests (folded into Academics), Activities/Achievements (folded into Profile), Shortlist/Applications/Offers (folded into one Universities pipeline), Recommendations and Transcript Requests (shown as status strips inside an Application, not standalone tabs, until Post-MVP).

## 5. Entity Relationship Map (Deliverable 5)

### 5.1 Object groups

**Foundation:** School, Academic Year, User, Role, Student, Counsellor Assignment
**Student record:** Student Profile, Subject, Grade, Predicted Grade, Test, Activity, Interest
**Journey execution:** Roadmap, Roadmap Template, Milestone, Task, Meeting, Meeting Note, Private Note, Shared Note
**Application pipeline:** University, Course, Shortlist Entry, Application, Application Round, Application Requirement, Document, Recommendation Request, Recommender, Transcript Request
**Outcome & oversight:** Offer, Scholarship, Final Decision, Risk, Intervention, Notification, Comment, Audit History

### 5.2 Key relationships and cardinality

- One **School** has many **Students**, **Users**, **Academic Years**.
- One **Student** has many **Counsellor Assignments over time** — modeled as a history table (not a single field), so reassigning a student preserves who was responsible for what and when. *(This resolves Open Question #1 from the PRD.)*
- One **Student** has one active **Roadmap**, cloned at assignment time from a **Roadmap Template** (school-level). The roadmap spans both Grade 11 and Grade 12 as one continuous record, organised into phases — it is not two separate objects that hand off at the grade transition.
- One **Roadmap** has many **Milestones**; one **Milestone** has many **Tasks**.
- **Task** is a single model with a polymorphic parent: a Task may belong to a Milestone, or directly to a Student (ad hoc), or to an Application Requirement, or to a Meeting (as a follow-up). This is what lets the same task appear on a Roadmap, inside an Application, and in a portfolio-wide queue without being three different objects.
- One **Student** has many **Meetings**; each Meeting may have one Shared Note, and separately, Private Notes exist as their own type (counsellor-only), linked to a Student and optionally to a Meeting.
- **University** has many **Courses**. A **Shortlist Entry** belongs to one Student and references one University+Course pair (plus an Application Round). Multiple Shortlist Entries can reference the same University if different Courses or Rounds are involved — this is how a student ends up applying to two courses at one university.
- A **Shortlist Entry** may convert into **at most one Application** (0-or-1 relationship). Before conversion, no Application exists; after conversion, the Shortlist Entry and Application stay linked so shortlist history remains visible from the application.
- One **Application** has many **Application Requirements** (essay, test score, recommendation, transcript, portfolio, interview).
- An **Application Requirement** may reference one or more **Documents**, and a **Document** may be referenced by multiple Requirements/Applications — documents are stored once in the student's library and linked, never duplicated. *(Resolves the document-reuse requirement.)*
- A **Recommendation Request** belongs to a Student (not to a single Application) and references one **Recommender**; it is linked to many Applications it supports. This is what lets one counsellor query "every application blocked by a missing recommendation" — the block is on the shared Recommendation Request, not duplicated per application.
- A **Transcript Request** follows the same pattern: belongs to a Student, links to the Applications it supports, has its own status pipeline.
- A **Risk** belongs to a Student, has a type/severity/status, and may have many **Interventions** logged against it over time.
- An **Application** may result in an **Offer** (0-or-1); an Offer may reference a **Scholarship**. Once a Student accepts one Offer, a single **Final Decision** record is created at the Student level, referencing that Application/Offer.
- **Comment** and **Notification** are both polymorphic/cross-cutting: a Comment can attach to a Task, Application, Shortlist Entry, or Document; a Notification (Post-MVP) references whatever object triggered it.
- **Audit History** is system-generated on writes to any of the above — not a user-created object.

*(A visual box-and-line version of this map is in the HTML file, §5.)*

## 6. Roadmap Architecture

- **Templates** are managed at the school level, under School Settings → Templates, by an Administrator (or Head of Counselling).
- A roadmap is **assigned** by cloning a template into a specific student's workspace; a counsellor can bulk-assign the same template to their entire caseload in one action (per PRD requirement).
- Once assigned, the roadmap is **customised freely** per student — the template is a starting point, not a constraint.
- Milestones contain tasks; **progress is calculated** as completed tasks / total tasks within a milestone, rolled up to overall roadmap %.
- **Grade 11 → Grade 12 is a phase transition within the same roadmap**, not a new object — this preserves full history across the transition without a re-onboarding step.
- Country-specific requirements (e.g., UCAS vs. Common App timelines) are **[ASSUMPTION]** handled at the template level in Post-MVP (multiple templates per country/system); in V1, with one school and manual entry, this is simply counsellor judgment when customising a student's roadmap.
- Delayed milestones surface as a **filter on Home/Applications**, not a separate page.
- A student's "next action" is simply the earliest-due, not-yet-complete task across their roadmap — surfaced directly on their Home screen.

**Placement decision:** Roadmaps are **both** — templates live at the school level, assigned/active roadmaps live inside each Student Workspace. There is no standalone "Roadmaps" top-level browsing page, because a roadmap only has meaning in the context of one student.

## 7. Task Architecture

One task model, five ways to view it:

- **Owner** (who's accountable) and **assignee** (who does the work) can differ — e.g., a counsellor owns "review Ananya's essay draft" but the task exists because the student uploaded it.
- **Status:** not started / in progress / submitted / needs revision / complete.
- **Priority:** inferred primarily from due-date proximity and whether it blocks an application; manual override allowed.
- **Due date, dependencies:** a task can depend on a milestone or another task (e.g., "submit essay" depends on "complete essay outline").
- **Comments and files:** attach directly to the task (comments reuse the polymorphic Comment object; files reference the student's Document library, not a task-specific upload).
- **Approvals:** a task tied to a submission (essay, document) has a review/approval step by the counsellor before being marked complete.

**Views into the same model:**
1. Across all students — the Applications Command Centre and Home's "overdue tasks" queue.
2. Inside a student workspace — the Roadmap tab.
3. Inside an application — as that application's requirement checklist.
4. Inside a roadmap — grouped under milestones.
5. On a dashboard — filtered to "mine, overdue, due soon."

No separate "Recommendation task" or "Transcript task" object exists — a recommendation or transcript follow-up is simply a Task whose parent is a Recommendation Request or Transcript Request.

## 8. Shortlist and Application Architecture

**Primary object:** the **University + Course combination**, held by a Shortlist Entry. This (not University alone, and not Application alone) is the right granularity, because it's the only level that correctly handles "two courses, one university" without duplicating university data, and correctly handles "considered but never applied" (a Shortlist Entry with no linked Application is a completely valid, common end state).

**Pipeline:**
`Suggested → Shortlisted → Approved → (student/counsellor decides to apply) → Application created → Submitted → Outcome`

- Suggested/Shortlisted/Approved are **states of the Shortlist Entry** (student can suggest; counsellor approves/categorises/flags).
- The transition to "Application" is a **deliberate, explicit action** (2 clicks, pulling fields from the Shortlist Entry — no re-entry), not an automatic status change.
- **Different application platforms** (Common App, UCAS, direct portal) are a field on the Application, not a structural difference — the platform doesn't need separate objects per platform in V1 (this was a PRD open item resolved simply here: track it as metadata).
- **Different deadlines per course, same university in different rounds:** both handled because Shortlist Entry (and its resulting Application) is scoped to University+Course+Round, not just University.
- **Shortlist changes over time:** per the PRD, only the current state is stored in V1 (no diff/reason log) — flagged again as a risk in §9, since it conflicts with the "everything should have visible history" principle used everywhere else in the product.
- **Counsellor/student disagreement:** the Shortlist Entry carries both a student preference field and a counsellor recommendation/category field side by side — disagreement is visible, not hidden, without needing a separate "dispute" object.
- **Parent input:** **[ASSUMPTION, Phase 2]** recorded as a comment on the Shortlist Entry, visible to the counsellor, but does not alter status — parents inform, they don't control.

## 9. Notes and Communication Architecture

Four distinct note/communication types, never merged into one generic field:

| Type | Where it appears | Who sees it | Linked to | Generates follow-ups? | In Activity History? |
|---|---|---|---|---|---|
| **Private counsellor note** | Meetings & Notes tab, private sub-view | Counsellor, Head of Counselling only | Student, optionally a Meeting | Yes (privately) | Yes, but entry is masked/summarised for non-counsellor viewers |
| **Shared meeting note** | Meetings & Notes tab | Counsellor + Student | Meeting | Yes | Yes |
| **Student update** | Meetings & Notes tab (student-authored) | Counsellor + Student | Student directly | Yes, if counsellor acts on it | Yes |
| **Task/Application comment** | Inline on the specific Task or Application | Whoever has access to that Task/Application | Task or Application (polymorphic Comment) | No (it *is* the follow-up mechanism) | Yes |

Teacher communication and Parent communication are **not yet designed** — both are Post-MVP roles, and the PRD leaves confidentiality mechanics for recommendations unresolved (Open Question #3). Recommendation: build these as narrow, purpose-specific channels (e.g., a Recommender only ever sees the Recommendation Request thread, never the student's Meetings & Notes tab) rather than extending the four types above to new roles.

Notes are **not exportable** in V1, matching the PRD's explicit non-goal.

## 10. Documents Architecture

- **One document library per student.** A document is uploaded once, categorised (essay, transcript, resume, portfolio, financial, certificate, application form, other), and then **linked** — not copied — to whatever it supports (a Task, an Application Requirement, an Offer, a Meeting).
- **Essays are a document category**, not a separate object — this is what makes reuse across applications (the same personal statement, linked to five applications) work without duplication.
- **Version history and review status** (draft / under review / feedback given / final) live on the Document itself, so feedback from one review is visible everywhere the document is linked.
- **Confidentiality:** a document can be flagged confidential (e.g., a final recommendation letter) — confidential documents are excluded from the student-visible library view even though the underlying storage is the same table.
- **Organisation is contextual, not folder-based** — the primary way to find a document is through the object it supports (this application's essay requirement) or through the library's category/search filters, not a manual folder tree. **[ASSUMPTION]** No nested folder system in V1.
- **Expiry** (e.g., a transcript that's only valid for one admissions cycle) — **[Open Question, Post-MVP]**, not addressed in the PRD; flagged in §12.

## 11. Search, Filters, and Saved Views

- **Global search** (top nav, all roles): finds students, universities, and — for counsellors — documents, by name/keyword.
- **Student search/filters** (on the Students portfolio): by name, grade, assigned counsellor, intended major, country, university, application round, deadline proximity, application status, risk level, task status, profile completion %, predicted grade, recent activity (or lack of it), missing document, recommendation status, transcript status.
- **University search** (on the Universities library): by name, country, course.
- **Saved views**: a counsellor can save a filter combination (e.g., "my G12 students with a deadline in 7 days") as a named view, reusable without rebuilding the filter. **Shared views** (a view any counsellor on the team can use, created by the Head of Counselling) are Post-MVP.
- **Recently viewed students**: a short list on Home and in global search, so returning to a student mid-task doesn't require re-searching.
- **Sorting**: standard ascending/descending on any filterable column in a list.

## 12. Notifications and Activity

Five distinct concepts, deliberately not collapsed into one:

- **Notification** (Post-MVP): a system-generated, dismissible prompt tied to a rule (e.g., deadline in 3 days). Excluded from V1 per the PRD's explicit non-goal.
- **Reminder** (Post-MVP): a scheduled nudge, e.g., "you haven't logged a meeting with this student in 60 days." Also excluded from V1.
- **Alert / Risk**: a flagged condition on a student (falling grades, no recent contact, application blocked) — this **is** in V1, but shown only as a queue item on Home/Students/Applications, never pushed as an email or push notification.
- **Activity log**: every meaningful edit, always recorded, always visible in a student's Activity History tab — this is passive and always-on, unlike a Notification which is active and dismissible.
- **Assigned task**: not a notification at all — it's a to-do item that lives in its normal place (Roadmap, Application, task queue) and is *found*, not pushed.

**V1 rule of thumb:** if it would require a push/email delivery mechanism, it's Post-MVP. If it's something a counsellor sees by simply opening the app, it's in scope.

### 13.2 Where role-based permissions aren't enough

- **Private counsellor notes** require **field/record-level** restriction, not just role-based — a Student role can never see this field regardless of which student record they're viewing (own or otherwise).
- **Recommendation final letter** needs the same treatment once built — visible to the Recommender and Counsellor, structurally invisible to the Student, even though the Student can see everything else about that same Application.
- **Cross-counsellor visibility** is currently flat (any counsellor sees any student) by PRD decision — this works at 3 counsellors but is a **record-level gap** the moment the team grows past a size where blanket visibility stops being appropriate (see Risks).
- **Parent visibility** will need field-level filtering on the *same* underlying objects (Shortlist Entry, Application) rather than a separate parent-facing copy of the data, to avoid the duplication risk described in §14.

## 15. Risks and Open Questions (Deliverable 8)

### Architecture risks
- The **Applications Command Centre** is the single most valuable screen in the product but is also the most complex to build correctly (it must aggregate across roadmap tasks, requirements, recommendations, and transcripts). Under-scoping it risks losing the platform's core differentiation vs. spreadsheets.
- Folding Recommendations and Transcript Requests into "status strips" inside Applications is right for V1 simplicity, but if either process becomes heavy in practice, this compression will need to be undone into full tabs — worth deciding trigger conditions now rather than reactively.

### Permission risks
- Flat "all counsellors see all students" works today (team of 3) but has no natural stopping point defined — decide the team-size or event (e.g., a second school, a larger team) that triggers a move to record-level restriction.
- Confidentiality enforcement for recommendation letters is unresolved at the PRD level; do not begin building the Documents or Recommendation objects until this is settled, or a rebuild is likely.

### Data duplication risks
- Documents must be modeled as **link, not copy**, from day one — retrofitting reuse-without-duplication after documents are already copied per-application is expensive.
- Grades/predicted grades live in Academics but are also referenced by shortlist eligibility — these should be one source of truth read in two places, not two data entry points.

### Navigation risks
- If Tasks, Roadmaps, or Shortlists are ever promoted back to top-level nav items (a common temptation as more filters get requested), the product risks fragmenting back into a "collection of modules" — see §16 for the reasoning to resist this.
- Teachers and Parents must never be given a "lite version of the counsellor nav" — that pattern silently expands what they can technically discover, even if individual pages are permission-checked.

### Scalability risks
- Manual entry for grades, universities, and student records will not scale past a handful of schools — this is an accepted, explicit V1 trade-off, not an oversight, but it is a hard ceiling on growth without integration work.
- One continuous Roadmap object per student (rather than one per grade) is the right call for V1's single curriculum, but multi-curriculum/multi-country schools will likely need country-specific template variants — not yet designed.

### Product-scope risks
- Because the platform stores highly sensitive private notes and (eventually) financial/scholarship information, scope creep toward "generic school CRM" is a real risk — every new feature request should be tested against "does this serve the counsellor's daily questions in §0 of the brief," not "could a school find this useful for something else."
- No monitoring for whether the product is actually beating spreadsheets exists yet (see PRD Open Question #11 on measuring time saved) — worth instrumenting from day one even informally.

### Questions that must be answered before user-flow design begins
1. What is the exact confidentiality enforcement mechanism for recommendation letters? (Blocks Recommendations + Documents flow design.)
2. Is shortlist/grade change history (diff + reason) coming to V1 or staying deferred? (Blocks Universities tab and Academics tab flow design — a "why did this change" affordance either exists or doesn't.)
3. Does the Applications Command Centre need real-time-feeling updates, or is daily/nightly refresh (per the PRD) acceptable for a screen this central to the daily workflow?
4. Will a second school be onboarded before V1 ships? (Determines whether School Settings needs a genuine multi-tenant config UI now, or can stay founder-managed a while longer.)
5. What exactly happens to "Assigned to" history when a student is reassigned — is a simple current-value field acceptable for the pilot, or does the audit-trail version need to exist from day one?

## 16. Critical Review (self-critique, and what changed as a result)

**Top-level items that were cut after review:** the brief's candidate list included Tasks, Roadmaps, Universities-as-shortlist, Shortlists, Applications, Documents, Meetings, Recommendations, and Transcript Requests as possible top-level nav items — nine candidates. Giving each one a permanent nav slot would produce a thirteen-item counsellor nav once Dashboard, Students, Reports, Templates, Settings are added — unmanageable, and worse, it would re-fragment the product into the same "separate modules" problem the brief explicitly warns against. Only **Applications** earned a standalone top-level slot, because it is the one cross-student queue with clear, standing daily value; everything else either lives inside the Student Workspace (because it only makes sense in the context of one student) or as a filter within Home/Students (because it's a query, not a destination).

**Overlap identified and resolved:** Shortlist, Applications, and Offers/Decisions were three separate candidate sections describing three stages of literally the same object lineage (Shortlist Entry → Application → Offer). Keeping them as three tabs would have meant the same University+Course pair appearing in three different places as it progressed — confusing, and a duplication risk. Merged into one **Universities** tab with an internal pipeline view.

**Same data appearing twice — caught and fixed:** Tests were a candidate standalone tab, but test scores are also academic performance data used in the same eligibility judgments as grades — splitting them risked two disconnected "how is this student doing academically" views. Folded into **Academics**. Similarly, Activities/Achievements were a candidate tab but are properly part of who-the-student-is, not part of the application-cycle machinery — folded into **Profile**.

**Where users might get confused:** having both a "Recommendation status" (visible to student) and a "Recommendation final letter" (invisible to student) on the same object could look like a bug ("why can't I open this") rather than a deliberate confidentiality boundary — the UI needs to visibly communicate *why* something is locked, not just hide it silently. Flagged for user-flow design, not solved here.

**Features that are filters, not pages:** "at-risk students," "students not recently contacted," "overdue tasks," "essays awaiting review," "missing documents" all appeared as candidate standalone sections in the brief. All of these are **filtered views of Students or Applications**, not separate data or separate pages — building them as pages would mean maintaining N slightly-different versions of the same student list.

**Actions moved into context rather than nav:** "add a note," "assign a task," "convert shortlist to application," "request a transcript" all read like they could be top-level actions or pages; all are implemented as **contextual actions inside the Student Workspace** (a button on the relevant tab), not destinations you navigate to independently.

**Where the student workspace still risks too many tabs:** 8 tabs is manageable, but Universities (shortlist + applications + offers) is doing the most work of any single tab. If application volume per student grows (many students applying to 10+ schools), this may need its own two-level nav (list + detail) rather than a single scrolling view — noted for user-flow design rather than solved here, since it's a layout decision, not a structural one.

**Administrative complexity deliberately delayed:** Integrations, Risk Rules, Notification Rules, and Data Import/Export are all real School Settings needs eventually, but none of them matter for a single pilot school with manual entry and no notifications — building the configuration UI for them now would be configuration for a feature that doesn't exist yet. Delayed to Post-MVP / Future.

**Where this could accidentally become a generic school CRM:** the original brief's full object list (School, Academic Year, Curricula, Grading Systems, etc.) could easily expand into a general school information system if every configuration option is exposed. The guardrail applied throughout this document: if a feature doesn't serve one of the counsellor's daily questions (§0) or the student's "what's next" question, it does not get a permanent place in the nav, regardless of whether some school administrator might theoretically want it.

**Where the architecture must clearly outperform spreadsheets and email, or it isn't worth building:** the Applications Command Centre and the unified student Activity History are the two areas doing that job — a spreadsheet cannot show "what changed since I last looked" or "everything blocked across my whole caseload, live" without manual maintenance. Every other section (Profile, Academics) is necessary but is not, by itself, a reason a counsellor would switch tools — it's the aggregation and cross-student views that justify the platform's existence, and they were treated as first-class in this revision rather than an afterthought.

## 18. Recommendations for User-Flow Creation

1. Start user-flow design with the **Applications Command Centre** and the **Universities tab** (shortlist → application conversion) — these are both the highest-value and highest-complexity flows, and getting the interaction model right here (queue → drill-in → action → back to queue) will set the pattern reused everywhere else.
2. Resolve **Open Question #1 (recommendation confidentiality mechanism)** before designing any Recommendation or Document flow — the UI treatment (locked icon vs. simply absent) depends entirely on the answer.
3. Design the **Meetings & Notes** flow around the actual physical rhythm described in the PRD — notes typed live, then revisited/edited after — rather than assuming a single "save once" interaction.
4. Treat **Home** as the last flow to finalize, not the first — it's a composite of every other flow's "needs attention" state, so it will keep changing shape until the underlying queues (tasks, risks, applications) are stable.
5. Prototype the **Universities tab** at realistic scale (a student with 8–12 shortlist entries in varying states) early, since §16 already flagged this as the tab most likely to need a two-level (list + detail) layout rather than a single scrolling page.
